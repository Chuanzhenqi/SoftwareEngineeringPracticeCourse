# 项目总结

项目名称：**Easy Project**

项目时间：2023/8/24 至 2023/9/1（共9日）

# 一、项目成果概述

我们组开发的Easy Project 是一个具备多人协同编辑、团队管理和聊天功能的 Web 项目。它旨在提供一个易于使用且高效的协作平台，使团队成员能够在一个统一的环境中协同工作、管理项目和进行实时沟通。

## （1）功能特点

以下是 Easy Project 提供的主要功能特点：

1. 多人协同编辑

•       提供实时协同编辑功能，多个用户可以同时编辑同一个文档或项目。

•       支持实时同步和冲突解决，确保数据一致性和可靠性。

•       提供丰富的编辑工具和格式选项，方便用户进行编辑和格式化操作。

1. 团队管理

•       提供团队和成员管理功能，方便创建和管理团队。

•       允许分配任务、设置优先级和截止日期，以及跟踪任务的进度和状态。

•       提供团队日历、里程碑和进度报告等功能，帮助团队成员了解项目进展和时间安排。

1. 实时聊天

•       提供实时聊天室或频道，方便团队成员之间进行即时沟通。

•       支持私聊和群组聊天，保证团队内部的有效沟通和协作。

## （2）主要技术栈及技术实现

以下是 Easy Project 使用的主要技术和工具：

•       **前端技术:** 使用现代的前端框架Vue3开发用户界面。UI方面使用Elemenet plus以及少量原生js（animejs）来辅助实现动画功能，文档编辑功能方面使用了Quill富文本编辑器，聊天功能使用腾讯的即时通信IM框架。

•       **后端技术:** 使用后端框架Spring构建服务器端应用程序，处理业务逻辑和数据存储。在实现多人协同的方面，利用了Nodejs搭建Websocket。

•       **数据存储:** 使用关系型数据库MySql存储和管理项目数据。在此基础上，后端使用集成的Jpa框架。

•       **实时通信:** 使用WebSocket实现实时通信和数据同步功能。

•       **身份验证和权限管理:** 使用JWT进行身份验证和授权。后端集成Spring Security，对用户的个人隐私进行完全加密处理，确保用户身份验证和权限管理。

## （3）项目成果及未来发展

我们成功实现了以下成果：

•       开发出一个功能齐全、易于使用的多人协同编辑系统，具备实时同步和冲突解决功能。

•       实现了团队管理模块，包括团队和成员管理、任务分配和进度跟踪等功能。

•       构建了实时聊天功能，支持团队内部的即时沟通和协作。

•       提供了用户友好的界面设计和交互体验，使用户能够轻松使用系统的各项功能。

我们计划在未来的发展中继续改进和增强系统的功能和性能。以下是一些可能的改进方向：

•       引入更多协同编辑功能，如批注、版本控制和协同筛选等。

•       增强团队管理模块，如项目计划、资源管理和报告生成等。

•       扩展聊天功能，支持文件共享、在线会议和通知提醒等功能。

•       提供移动端应用程序，方便用户在移动设备上随时随地访问和使用系统。

# 二、项目反思

虽然结果向好，但在实际实现项目的过程中，我们踩坑无数，阻碍重重。特以志之，以作前车之鉴，用以鞭策未来的软件工程项目，万望勿重蹈覆辙。

## （1）软工团队管理合作方面

### 阿里云平台的使用

团队管理方面并非乏善可陈，而是可以有更大的提升空间。

**阿里云效平台**是个很好的项目管理平台，我认为，本次项目管理可圈可点的方面正是得益于此平台的帮助。我们的项目文档、项目代码、工作进度可以统一管理，做到了项目的统一化。

提升空间则是**对阿里云效平台的利用不够。**比如”流水线”功能，我认为是个会给项目带来巨大提升的好功能，但是我们本次没有利用。

除此之外，我们其实并非完全按照云平台来按部就班地推进项目。我们并没有做到*项目计划推动项目的真实进展*，而是*项目真实进展左右着项目计划的安排*。这是我们需要改善的最大一点。

记得乔布斯提出一个概念：

消费者并不知道自己需要什么，直到我们拿出自己的产品，他们就发现，这是我要的东西。

对于软件开发者来说，我们确实不知道自己需要什么样的产品来辅助开发。而通过这次的体验经历来看，阿里云效是个较为理想的产品。以后应该多多探索其功能，以此加强开发效率。

### Apifox的使用

对Apifox的使用不足，也是我们需要提升的重要一点。

我们后端以前使用postman等Api测试框架。转到Apifox之后，发现它基本满足后端的测试功能，但是并没有最大化其利用，导致之后前后端对接口时花费了大量时间。进行反思之后，我们将原因归结到Apifox的使用不充分上。

以后的项目应该遵循以下步骤：

1. 项目初期，前端和后端共同敲定接口规范，在Apifox上搭建最基本的接口模型。
2. 项目中期，前端和后端完全分离开发。**对于前端，充分利用****Apifox****的mock****功能（而不是找后端开服务器，进行测试）；对于后端，只需要遵守接口规范，便可以在结项时完美对接。**

## （2）框架选择方面

对框架的测试度不够，是我们这次项目最大的问题。纵使框架再优秀、用户群体再庞大，也难免会出现兼容性的bug或者难以发现的底层bug。我们这次使用的Quill框架、腾讯IM即时通信框架，甚至是Spring Security框架和前端Vue3框架，都出现了或多或少的问题。

以下列举几个遇到的问题：

•       **Quill**底层出现bug，并且有人于2016年已经在官方Github上提相关issue，至今未解决。最后我们将其归结为与vue3框架的兼容性bug。

•       **Vue3**前端项目最终进行部署时，报错某个css文件异常，无法通过build构建，且报错信息非常笼统，根本无法定位到具体的文件。

将来对于框架的选择上，我们应该**参考快速原型模型**的软件工程方法，先广撒网，选择尽可能多的框架，再粗略评测每个框架的优劣和兼容性，而不是完全听信博客的测评和推荐。**框架兴许确实不错，但兼容项目代码、拟合项目需求的框架，才是最合适的框架。**

## （3）系统架构方面

我们本次项目中，前端和后端各进行了一次重构。

后端的重构，使得项目的鲁棒性和安全性提升了，架构也更加清楚。前端则是将UI界面大改为统一的Element Plus框架。

纵使有一定价值，但是重构确实花费了巨大的时间和人力开销。其实这两点都是可以从一开始就避免的。项目伊始，前端就应该敲定项目UI界面的风格而不是之后更改，后端也应该大致分出软件包、工具类，画出类图，进行需求分析。

# 三、总结

总体来说，虽然项目结束得不错，但是可以做得更好。我们加深了对框架的熟练度，更重要的是我们发现了自身存在的相关问题，这对我们未来的合作具有重要的启示。

1. 我们在项目中展现了出色的团队合作和协作能力。我们相互支持，共同追求目标，并且能够高效地分工合作。这种团队精神是我们成功完成项目的关键之一。
2. 在项目的过程中，我们也意识到了自身的一些不足之处。例如，我们在沟通和协调方面可能存在一些挑战，导致信息传递不够及时和准确。这给项目进展带来了一些不必要的延迟和误解。我们需要更加注重沟通的准确性和及时性，以避免类似的问题。
3. 我们的需求分析和规划工作可能没有做得充分。这导致了一些需求的变更和补充，给项目进度和质量带来了一些压力。为了避免类似情况的发生，我们需要在项目启动阶段更加仔细地进行需求分析和规划，确保项目的基础工作做得更加扎实。

总的来说，这次项目是一个宝贵的学习机会。我们不仅加强了对框架的掌握，还发现了自身存在的问题，并且从中吸取了宝贵的经验教训。这将使我们在未来的项目中更加成熟和专业。我们要牢记这次项目中的经验，不断改进自己，为下一次的挑战做好准备。
