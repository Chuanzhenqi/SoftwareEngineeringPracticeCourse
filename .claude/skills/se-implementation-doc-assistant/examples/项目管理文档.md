# 软件开发计划书 - 第14组

## 1. 引言

### 1.1 编写目的
本文档旨在为“风行旅途”在线火车购票旅游系统的开发提供详细的计划指导，明确项目目标、范围、进度、资源分配及风险管理策略。预期读者包括项目组成员、项目管理人员、客户代表及相关利益相关方。

### 1.2 背景
*   **项目名称**：风行旅途在线火车购票旅游系统
*   **项目组别**：第14组
*   **任务提出者**：远方开发团队
*   **开发者**：第14组开发团队
*   **用户**：需要购买火车票、预订酒店及订购火车餐的旅客
*   **项目意义**：解决传统购票流程繁琐、信息分散的问题，提供一站式旅行服务体验。

## 2. 项目概述

### 2.1 项目目标
1.  **功能目标**：在6个月内完成包含火车购票、酒店预订、火车餐购买、订单管理及支付功能的系统上线。
2.  **质量目标**：系统上线后严重缺陷率为0，一般缺陷率低于5个/千行代码，系统可用性达到99.9%。
3.  **成本目标**：项目开发成本控制在预算范围内，人力资源利用率达到85%以上。
4.  **进度目标**：严格按照里程碑计划执行，确保各阶段交付物按时提交。

### 2.2 项目范围
**包含内容**：
*   用户注册与登录模块
*   火车车次查询与选票模块
*   酒店搜索与预订模块
*   火车餐浏览与订购模块
*   购物车与订单管理模块
*   支付接口集成模块
*   消息通知模块
*   后台管理模块（基础版）

**不包含内容**：
*   机票预订功能（二期规划）
*   旅游景点门票预订（二期规划）
*   复杂的会员积分体系（简化版一期上线）
*   多语言支持（仅支持中文）

### 2.3 项目约束
1.  **时间约束**：必须在202X年X月X日前完成上线，以配合暑期旅游高峰。
2.  **技术约束**：必须采用Java Spring Boot后端和Vue.js前端技术栈。
3.  **资源约束**：开发团队固定为8人，不可随意增加人员。
4.  **外部依赖**：依赖第三方铁路数据接口和支付网关的稳定性。

## 3. 项目组织与人员分工

### 3.1 组织结构图文字描述
项目组采用敏捷开发模式，组织结构如下：
*   **项目经理 (PM)**：1名，负责整体规划、进度控制、风险管理和对外沟通。
*   **产品负责人 (PO)**：1名（兼任），负责需求梳理、优先级排序和验收测试。
*   **架构师**：1名，负责系统架构设计、技术选型和核心代码评审。
*   **后端开发组**：3名，负责微服务开发、数据库设计和接口实现。
*   **前端开发组**：2名，负责Web页面开发、交互实现和移动端适配。
*   **测试工程师 (QA)**：1名，负责测试用例编写、执行测试和缺陷跟踪。

### 3.2 人员职责分工表

| 角色 | 人数 | 主要职责 | 关键技能要求 |
| :--- | :--- | :--- | :--- |
| 项目经理 | 1 | 制定计划、监控进度、协调资源、风险管理 | PMP认证、沟通能力、敏捷管理 |
| 产品负责人 | 1 | 需求分析、原型设计、用户故事编写、验收 | 业务理解、Axure/Sketch、沟通 |
| 架构师 | 1 | 架构设计、技术难点攻关、代码规范制定 | 微服务架构、高并发设计、Java |
| 后端开发 | 3 | 业务逻辑实现、API开发、数据库操作 | Spring Boot, MySQL, Redis, RabbitMQ |
| 前端开发 | 2 | 界面开发、交互逻辑、性能优化 | Vue.js, Element UI, CSS3, JavaScript |
| 测试工程师 | 1 | 测试计划、用例执行、自动化测试、质量报告 | JUnit, Selenium, JMeter, Bug管理 |

## 4. 项目进度计划

### 4.1 里程碑计划
项目总周期预计为24周（约6个月），主要里程碑如下：

1.  **M1：项目启动与需求确认**（第2周结束）
    *   交付物：项目章程、需求规格说明书、原型图
2.  **M2：系统设计完成**（第4周结束）
    *   交付物：概要设计说明书、详细设计说明书、数据库设计文档
3.  **M3：核心功能开发完成（Alpha版）**（第12周结束）
    *   交付物：可运行的核心功能系统（购票、下单）、单元测试报告
4.  **M4：系统集成与测试完成（Beta版）**（第18周结束）
    *   交付物：完整功能系统、集成测试报告、性能测试报告
5.  **M5：用户验收测试通过（RC版）**（第22周结束）
    *   交付物：UAT验收报告、用户手册、部署文档
6.  **M6：正式上线**（第24周结束）
    *   交付物：上线系统、运维手册、项目总结报告

## 5. 资源计划

### 5.1 人力资源计划
*   **投入方式**：全职投入，除节假日外无其他假期。
*   **技能提升**：项目初期安排2天技术培训和1天敏捷开发流程培训。
*   **备份机制**：关键岗位（如架构师、核心后端）设立AB角，防止人员流动风险。

### 5.2 硬件资源计划

## 6. 风险管理计划

### 6.1 风险识别与评估

| 风险ID | 风险描述 | 可能性 | 影响程度 | 风险等级 | 应对策略 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| R01 | 第三方铁路接口不稳定或文档缺失 | 中 | 高 | 高 | 提前联系接口方，建立模拟桩代码，制定降级方案 |
| R02 | 核心开发人员离职 | 低 | 高 | 中 | 实施代码审查制度，文档实时更新，建立AB角机制 |
| R03 | 需求变更频繁导致进度延误 | 高 | 中 | 高 | 严格执行变更控制流程，预留缓冲时间，采用敏捷迭代 |
| R04 | 性能不达标（高并发下响应慢） | 中 | 高 | 高 | 早期进行性能压测，引入缓存机制，优化数据库索引 |
| R05 | 支付接口申请审批滞后 | 中 | 中 | 中 | 项目启动即提交申请，准备备用支付渠道方案 |
| R06 | 测试环境数据不足 | 低 | 中 | 低 | 编写数据生成脚本，使用脱敏的生产数据副本 |

### 6.2 风险监控
*   **监控频率**：每周例会检查风险状态。
*   **触发机制**：当风险可能性或影响程度发生变化时，立即更新风险登记册。
*   **上报机制**：高风险事项直接上报项目指导委员会。

## 7. 质量保证计划

### 7.1 质量标准
*   代码规范遵循《阿里巴巴Java开发手册》。
*   单元测试覆盖率不低于80%。
*   严重及以上级别缺陷必须在上线前清零。
*   系统响应时间满足概要设计要求。

### 7.2 质量控制活动
1.  **代码审查 (Code Review)**：每个Sprint结束前进行，由架构师和资深开发主导。
2.  **持续集成 (CI)**：代码提交后自动触发构建和单元测试。
3.  **测试阶段**：
    *   单元测试：开发人员自测。
    *   集成测试：测试工程师执行，验证模块间接口。
    *   系统测试：全功能流程测试。
    *   性能测试：模拟高并发场景。
4.  **缺陷管理**：使用Jira跟踪缺陷生命周期，定期分析缺陷分布。

## 8. 沟通管理计划

### 8.1 沟通矩阵

| 沟通对象 | 沟通内容 | 频率 | 方式 | 负责人 |
| :--- | :--- | :--- | :--- | :--- |
| 项目组成员 | 任务进度、技术问题、每日站会 | 每日 | 面对面/视频会议 | 项目经理 |
| 项目指导委员会 | 里程碑汇报、重大风险、资源申请 | 每两周 | 会议 + PPT报告 | 项目经理 |
| 客户代表 | 需求确认、演示反馈、验收测试 | 每Sprint结束 | 演示会议 | 产品负责人 |
| 外部合作方 | 接口联调、技术对接 | 按需 | 邮件/即时通讯 | 架构师 |

### 8.2 会议制度
*   **每日站会**：15分钟，同步昨日进展、今日计划及阻碍。
*   **Sprint计划会**：每个迭代开始，确定迭代目标和任务。
*   **Sprint评审会**：每个迭代结束，演示成果，收集反馈。
*   **Sprint回顾会**：每个迭代结束，总结经验教训，改进流程。
