# SE Requirements Doc Assistant

将课程项目需求整理为可评审、可测试、可追踪的需求文档。

## Overview

将课程项目需求整理为可评审、可测试、可追踪的需求文档。

### 适用模板

优先使用：
- `assets/软件需求规格说明书.md`

若为夏季项目，则将需求内容映射到：
- 项目管理文档中的需求与迭代目标
- 项目总结中的“目标达成与问题复盘”

### 输入

- 项目背景、目标用户、业务场景
- 已有原型/页面/接口草图
- 约束（时间、技术、人员）
- 非功能目标（性能、安全、可用性）

### 输出要求

- 需求采用唯一编号（如 `FR-001`, `NFR-001`, `IF-001`）
- 每条需求必须可验证（含验收方式）
- 区分“已确认”与“待确认”
- 明确边界：范围内 / 范围外

## Workflow

### 1. 需求分层与提炼

- **功能需求 (FR)**：提炼触发条件、处理规则、成功输出、异常处理和验收标准。
- **非功能需求 (NFR)**：量化性能、安全、可用性和可维护性指标。

### 2. 接口与数据需求

- **外部接口**：定义用户接口、软件接口和通信接口。
- **内部数据**：梳理核心实体、字段约束、关系说明及其一致性策略。

### 3. 建立追踪链

构建追踪关系：
- 需求编号 → 设计模块
- 需求编号 → 测试用例
- 需求编号 → 实现证据

### 质量检查与生成的策略

- **不删除模板条目**：仅填充或标注“不适用”，确保文档结构完整。
- **量化指标**：避免“友好”“高效”等主白表述，确保需求可测试。
- **闭环管理**：对无法确认的信息，统一放入“待确认问题”列表。
