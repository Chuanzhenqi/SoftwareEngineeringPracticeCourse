## 3 功能模块设计

### 3.1 SS-01 用户认证模块

#### 3.1.1 模块概述

| 属性 | 内容 |
|------|------|
| 模块编号 | SS-01 |
| 模块名称 | 用户认证模块（UserAuthModule） |
| 输入 | 手机号、验证码/密码、角色信息 |
| 处理 | 验证码发送与校验、密码校验、JWT 生成、权限路由 |
| 输出 | JWT Token、用户基础信息 VO、错误码 |
| 依赖 | 短信服务（IF-006）、Redis（会话存储）、MySQL（users表） |
| 对应需求 | FR-001, FR-002, NFR-010, NFR-011 |

#### 3.1.2 处理流程

**注册流程（FR-001）**：

```
输入: phone, smsCode, password
  │
  ├─[1] 格式校验: phone 11位正则 ^1[3-9]\d{9}$
  │
  ├─[2] 查询 Redis: sms:code:{phone} 是否存在且匹配
  │       └─ 不匹配/不存在 → 返回 40001 验证码错误
  │
  ├─[3] 查询 MySQL: SELECT * FROM users WHERE phone=?
  │       └─ 已存在 → 返回 40901 手机号已注册
  │
  ├─[4] 密码加密: passwordHash = BCrypt.hash(password, cost=12)
  │
  ├─[5] 写入 MySQL: INSERT INTO users(phone, password_hash, role=1)
  │
  ├─[6] 删除 Redis 验证码: DEL sms:code:{phone}
  │
  └─[7] 生成 JWT → 返回 token + userVO
```

**登录流程（FR-002）**：

```
输入: phone, password  (或 phone, smsCode)
  │
  ├─[1] 查询失败次数: GET limit:login:fail:{phone}
  │       └─ ≥ 5次 → 返回 42901 账号已锁定，剩余时间X分钟
  │
  ├─[2] 查询 MySQL: SELECT * FROM users WHERE phone=? AND status=1
  │       └─ 不存在/封禁 → 返回 40401 账号不存在
  │
  ├─[3] 密码模式: BCrypt.verify(inputPwd, storedHash)
  │       └─ 失败 → INCR limit:login:fail:{phone}, TTL=30min
  │               → 返回 40001 密码错误，剩余N次
  │
  ├─[4] 生成 JWT: payload={userId, role, exp=7days}
  │
  ├─[5] 写入 Redis: SET session:token:{userId} → token, TTL=7days
  │
  └─[6] 重置失败计数: DEL limit:login:fail:{phone}
        → 返回 token + userVO
```

#### 3.1.3 算法描述

**JWT 生成算法**：
```
header  = Base64URL({ "alg": "HS256", "typ": "JWT" })
payload = Base64URL({
    "userId": <Long>,
    "role":   <1=学生|2=商家|3=骑手|4=管理员>,
    "iat":    <当前时间戳>,
    "exp":    <iat + 7*24*3600>
})
signature = HMAC_SHA256(header + "." + payload, SERVER_SECRET_KEY)
token = header + "." + payload + "." + signature
```

**权限拦截器（RBAC）**：
```
每次请求进入 AuthInterceptor:
  1. 从 Header 取 Authorization: Bearer <token>
  2. JWT.verify(token, SECRET_KEY) → 解析 userId, role
  3. 查询 Redis session:token:{userId} 验证 token 未被踢出
  4. 比对接口注解 @RequireRole(roles=[2]) 与实际 role
  5. 不匹配 → 返回 40102 权限不足
  6. 通过 → 将 userId/role 写入 ThreadLocal 供 Service 使用
```

#### 3.1.4 接口定义

**POST /api/v1/auth/sms/send** — 发送验证码

```json
成功响应(200): { "code": 20000, "message": "发送成功" }
```

### 3.2 SS-02 主页与推荐模块

| 属性 | 内容 |
|------|------|
| 模块编号 | SS-02 |
| 模块名称 | 主页与推荐模块（HomeRecommendModule） |
| 输入 | userId（已登录）或 null（游客）；搜索关键词 |
| 处理 | 推荐列表生成、关键词模糊查询、结果排序 |
| 输出 | 推荐店铺列表 VO、推荐商品列表 VO、搜索结果 VO |
| 依赖 | Redis（推荐缓存）、MySQL（shops/products 表全文索引） |
| 对应需求 | FR-010, FR-011 |

#### 3.2.1 推荐算法描述

```
推荐分 = 0.4 × 归一化销量 + 0.3 × 归一化评分 + 0.2 × 新鲜度衰减 + 0.1 × 个性化加权

其中：
  归一化销量   = sales_count / MAX(sales_count) over 全平台
  归一化评分   = (rating - 1.0) / 4.0
  新鲜度衰减   = exp(-λ × 上架天数)，λ=0.01
  个性化加权   =
      若用户有历史订单：该店铺/商品类别在用户历史中出现频次 / 总历史订单数
      若新用户：         默认 0（退化为纯热度推荐）

新用户冷启动：直接返回全平台 sales_count 前10名商品 + 评分前5名店铺
```

**搜索实现**：
- 使用 MySQL FULLTEXT 全文索引（products.name、shops.name）
- 查询语句：`SELECT * FROM products WHERE MATCH(name) AGAINST(? IN BOOLEAN MODE)`
- 结果按相关度排序（MySQL FULLTEXT 内置相关度评分）
- 搜索历史写入 Redis List：`LPUSH search:history:{userId} {keyword}`，`LTRIM` 保留20条

#### 3.2.2 接口定义

**GET /api/v1/home** — 获取主页推荐内容
```
Headers: Authorization: Bearer <token>（可选，无则返回热度榜）
成功响应:
{
  "code": 20000,
  "data": {
    "recommendShops":    [ { shopId, name, rating, deliveryTime, coverUrl } × 5+ ],
    "recommendProducts": [ { productId, name, price, shopName, imageUrl } × 10+ ]
  }
}
```

**GET /api/v1/search?keyword={kw}&type={shop|product|all}&page={n}&size={20}** — 搜索
```
成功响应:
{
  "code": 20000,
  "data": {
    "shops":    [ { shopId, name, rating, type, location } ],
    "products": [ { productId, name, price, shopId, shopName } ],
    "total":    42
  }
}
```

### 3.3 SS-03 店铺与商品模块

| 属性 | 内容 |
|------|------|
| 模块编号 | SS-03 |
| 模块名称 | 店铺与商品模块（ShopProductModule） |
| 输入 | 筛选条件（type/location/rating/priceRange/deliveryTime）、shopId、productId |
| 处理 | 多维筛选查询、详情查询、评价分页查询 |
| 输出 | 店铺列表 VO、店铺详情 VO、商品详情 VO |
| 依赖 | MySQL（shops/products/reviews 表）、Redis（详情缓存） |
| 对应需求 | FR-020, FR-021, FR-022 |

#### 3.3.1 店铺筛选算法描述

```
筛选条件（均为可选，多条件 AND 组合）：
  type         → WHERE type = ?
  minRating    → WHERE rating >= ?（如 4.0、4.5）
  priceRange   → WHERE min_order_amount BETWEEN ? AND ?
  deliveryTime → WHERE avg_delivery_time <= ?（如 30、45）
  location     → WHERE location LIKE CONCAT('%', ?, '%')

排序策略（默认综合排序）：
  综合排序 = 0.5 × 评分 + 0.3 × 销量 + 0.2 × (1 / 配送时间)
  用户可切换：按评分降序 / 按销量降序 / 按配送时间升序
```

#### 3.3.2 详情缓存策略

```
查询店铺详情（shopId）:
  1. GET Redis shop:detail:{shopId}
     命中 → 直接返回 JSON
  2. 未命中 → 查 MySQL：
       JOIN shops, products（热销前5）, reviews（评分汇总）
  3. 写入 Redis：SET shop:detail:{shopId} {json}, TTL=300s
  4. 返回结果

缓存失效触发：
  商家修改店铺/商品信息 → 主动 DEL shop:detail:{shopId}
  管理员下架操作        → 主动 DEL shop:detail:{shopId}
```

### 3.4 SS-04 购物车与下单模块

| 属性 | 内容 |
|------|------|
| 模块编号 | SS-04 |
| 模块名称 | 购物车与下单模块（CartOrderModule） |
| 输入 | userId、shopId、productId、数量、地址ID、couponId、支付方式 |
| 处理 | 购物车 CRUD、金额计算、优惠券校验、订单创建、支付发起 |
| 输出 | 购物车 VO、订单 VO、支付参数 VO |
| 依赖 | Redis（购物车存储）、MySQL（orders/order_items）、营销服务、支付 SDK |
| 对应需求 | FR-030, FR-031 |

#### 3.4.1 购物车设计

```
Redis 数据结构: Hash
Key:   cart:{userId}:{shopId}
Field: {productId}
Value: {quantity}
TTL:   7天（每次操作刷新）

操作:
  加入: HSET cart:{userId}:{shopId} {productId} {qty}
  修改: HSET cart:{userId}:{shopId} {productId} {newQty}
  删除: HDEL cart:{userId}:{shopId} {productId}
  查询: HGETALL cart:{userId}:{shopId}
        → 批量查询 products 表获取名称/价格/状态
        → 过滤 status=0 的下架商品（标记提示）

跨店限制:
  前端按 shopId 维度展示购物车，不同店铺购物车独立
  结算时仅允许单店铺商品提交
```

#### 3.4.2 下单流程（含事务边界）

```
输入验证:
  [1] 校验购物车非空，商品均在架，shop 为营业状态
  [2] 校验收货地址归属当前 userId
  [3] 计算原始金额 = SUM(unit_price × quantity)

优惠券校验（调用 SS-10）:
  [4] 若 couponId 非空:
        验证 user_coupons: status=0, expire_at>now, shop_id 匹配
        计算 discountAmount

金额计算（服务端强制）:
  [5] payAmount = totalAmount - discountAmount + deliveryFee
      payAmount = MAX(payAmount, 0.01)  // 最小0.01元

事务写入（MySQL @Transactional）:
  [6] INSERT orders(..., status=0)  → 获取 orderId
  [7] INSERT order_items 批量插入商品快照
  [8] 若使用优惠券: UPDATE user_coupons SET status=1, used_order_id=orderId

支付发起:
  [9] 调用支付 SDK 生成预付单（微信/支付宝）
  [10] 返回支付参数给客户端
  [11] 启动超时检查: 15分钟无支付 → 定时任务取消订单，回滚优惠券状态
```

#### 3.4.3 支付回调处理（幂等设计）

```
POST /api/v1/orders/payment/callback

[1] 验证签名（防篡改）: 使用支付平台提供的验签算法
[2] 幂等检查: SELECT status FROM orders WHERE order_id=?
      status=1（已支付）→ 直接返回 SUCCESS（防重复回调）
      status=6（已取消）→ 返回 FAIL，通知支付平台退款
      status=0（待支付）→ 继续处理
[3] 事务更新:
      UPDATE orders SET status=1, pay_time=now() WHERE order_id=? AND status=0
[4] 发布 MQ 消息: topic=ORDER_PAID, body={orderId, shopId, userId}
[5] 清空对应购物车: DEL cart:{userId}:{shopId}
[6] 返回 SUCCESS
```

#### 3.4.4 订单状态机

```
          超时取消(15min)
    ┌──────────────────────────────┐
    │                              ↓
[0]待支付 ──支付成功──> [1]已支付 ──商家接单──> [2]备餐中
```

#### 3.8.1 智能派单算法

```python
# 评分模型（伪代码）
def score_rider(rider, order):
    # 参数（初始值，运营后调优）
    α = 0.5  # 距离权重
    β = 0.3  # 当前负载权重
    γ = 0.2  # 历史评分权重

    # 距离：骑手到商家的直线距离（米），调用 Redis GEODIST
    dist = geodist(rider.location, order.shop_location)
    dist_score = 1.0 / (1.0 + dist / 1000)  # 距离越近得分越高

    # 当前负载：骑手当前承接的未完成订单数
    load = get_active_task_count(rider.id)
    load_score = 1.0 / (1.0 + load)

    # 历史评分（骑手被评价的平均分，1-5）
    hist_score = (rider.avg_rating - 1.0) / 4.0

    return α * dist_score + β * load_score + γ * hist_score

def smart_dispatch(order):
    # 获取附近3km内在线骑手（Redis GEO 范围查询）
    candidates = georadius("rider:location", order.shop_lng, order.shop_lat, 3000, "m")

    if not candidates:
        candidates = georadius(...)  # 扩大到5km

    # 按评分降序排列
    ranked = sorted(candidates, key=lambda r: score_rider(r, order), reverse=True)

    # 向评分最高骑手推送派单，等待10秒
    # 若拒绝/超时，推送下一位
    for rider in ranked[:3]:
        push_dispatch(rider, order)
        if wait_accept(rider, timeout=10s):
            assign(order, rider)
            return

    # 全部拒绝 → 放入抢单池
    publish_to_grab_pool(order)
```

#### 3.8.2 抢单并发控制

```
POST /api/v1/rider/orders/{taskId}/grab

[1] 分布式锁: SET lock:task:{taskId} {riderId} NX EX 5
      获锁失败 → 返回 40901 抢单失败，已被他人接走
[2] 查询 delivery_tasks: status=0（待接单）
      status≠0 → 释放锁 → 返回 40901
[3] 乐观锁更新:
      UPDATE delivery_tasks SET rider_id=?, status=1, assigned_at=now()
      WHERE task_id=? AND status=0
      影响行数=0 → 释放锁 → 返回 40901
[4] 释放分布式锁: DEL lock:task:{taskId}
[5] 推送通知给用户（订单已被接单）
[6] 返回 20000 抢单成功，配送信息
```

```

[2] 防超发（Redis 原子操作）:
```

#### 3.10.2 周榜生成算法

```
定时任务（每周一 02:00 执行）:

SELECT shop_id,
       COUNT(o.order_id)     AS order_count,
       AVG(r.rating)         AS avg_rating,
       SUM(CASE WHEN r.rating>=4 THEN 1 ELSE 0 END) / COUNT(r.review_id) AS praise_rate
FROM orders o
LEFT JOIN reviews r ON o.order_id = r.order_id AND r.target_type=2
WHERE o.created_at >= NOW() - INTERVAL 7 DAY AND o.status=4
GROUP BY shop_id

综合榜单分 = 0.5 × 归一化订单量 + 0.3 × 归一化评分 + 0.2 × 好评率
按综合分降序取前 20 名

写入 Redis: SET rank:weekly {json结果}, TTL=7days + 1hour
```

## 5 异常处理设计

### 5.1 出错信息管理

系统统一错误码规范（延续概要设计 6.1 节）：

| 错误码 | HTTP 状态 | 描述 | 触发场景 |
|--------|---------|------|---------|
| 20000 | 200 | 成功 | 正常响应 |
| 40001 | 400 | 参数错误/验证码错误 | 格式不合法、验证码不匹配 |
| 40101 | 401 | Token 过期 | JWT 过期或被踢出 |
| 40102 | 403 | 权限不足 | 角色不匹配 |
| 40401 | 404 | 资源不存在 | 商品/店铺/订单不存在 |
| 40901 | 409 | 业务冲突 | 商品已下架、券已使用、抢单失败 |
| 42901 | 429 | 请求过于频繁 | 超出限流阈值 |
| 50001 | 500 | 数据库异常 | DB 连接失败、事务回滚 |
| 50002 | 500 | 第三方服务异常 | 地图/支付/短信 API 故障 |
| 50003 | 500 | 系统内部异常 | 未预期异常 |

**全局异常处理器（Spring @ControllerAdvice）**：
```java
@ExceptionHandler(BusinessException.class)
→ 返回业务错误码与描述

@ExceptionHandler(MethodArgumentNotValidException.class)
→ 返回 40001 + 字段校验错误列表

@ExceptionHandler(Exception.class)
→ 记录 ERROR 日志（含 traceId）
→ 返回 50003 系统内部异常，不暴露详细堆栈给客户端
```

### 5.2 故障预防与补救

| 故障场景 | 预防措施 | 补救措施 |
|---------|---------|---------|
| 订单提交失败 | 数据库事务保证原子性 | 自动回滚，购物车内容保留；提示用户重试 |
| 支付回调丢失 | 定时任务（每5分钟）主动查询支付平台订单状态 | 查询确认已支付 → 补触发 ORDER_PAID 消息 |
| Redis 宕机 | Redis 持久化（AOF），主从备份 | 缓存降级：直接查 MySQL；购物车从 MySQL 备份表恢复 |
| MySQL 主库故障 | 主从复制，从库实时同步 | 自动切换从库为主库（人工或自动切换脚本） |
| MQ 消息丢失 | 消息持久化（MQ 落盘）+ 消费确认（ACK） | 未 ACK 消息自动重投，最多重试 3 次 |
| 第三方 API 超时 | 设置超时时间（地图3s/支付5s/短信3s） | 降级处理（见各模块降级策略）；告警运维 |
| 骑手位置上报中断 | 超过60秒无上报，系统自动标记"位置不可用" | 前端提示"骑手位置暂不可用"，联系电话不受影响 |
| 优惠券超发 | Redis DECR 原子操作 | 若发现超发（对账）→ 对额外发出的券补偿处理 |

### 5.3 系统维护设计

- **健康检查接口**：`GET /actuator/health`，返回各依赖组件（DB/Redis/MQ）状态
- **优雅关机**：接收 SIGTERM 信号后，等待当前请求处理完毕（最长30秒）再退出
- **滚动发布**：K8s Rolling Update，逐步替换 Pod，保证零停机
- **维护公告**：维护前通过 App Push 和主页 Banner 提前24小时通知用户
- **数据备份**：MySQL 每日全量备份 + 实时 binlog，备份保留30天；Redis AOF + RDB 双持久化
- **日志轮转**：日志文件按天切割，保留90天，超期自动归档到对象存储

## 6 性能优化和安全设计

### 6.1 性能优化

**缓存策略**（参照 RAG 检索到的往年样例"热门商品接口采用缓存优化，图片等静态资源使用 OSS 分发"）：

| 优化点 | 策略 | 预期收益 |
|--------|------|---------|
| 主页推荐接口 | Redis 缓存 5分钟（TTL=300s） | DB 压力降低 95%，响应 <50ms |
| 店铺/商品详情 | Redis 缓存 5分钟，写操作主动失效 | 详情页响应 <100ms |
| 购物车 | 全量存 Redis，MySQL 异步备份 | 购物车操作 <20ms |
| 商品图片/封面图 | CDN 分发（OSS + CDN 加速） | 图片加载 <500ms（全球） |
| 骑手位置 | Redis GEO，不写 MySQL | 位置查询 <5ms |
| 数据库读写分离 | 读请求走从库，写请求走主库 | 主库写 QPS 提升 50% |
| 全文搜索 | MySQL FULLTEXT 索引，后期可升级 ES | 搜索响应 <200ms |
| 数据库连接池 | HikariCP，最大连接数 50/服务 | 避免连接等待 |
| SQL N+1 问题 | 批量查询替代循环单查 | 接口响应降低 80% |

**高峰期并发保护**：

```
限流（Spring 拦截器 + Redis）:
  普通接口: 100 req/min / 用户
  下单接口: 10 req/min / 用户（防刷单）
  验证码接口: 5次/天 / 手机号

熔断（Resilience4j）:
  调用外部 API（地图/支付/短信）：
    失败率 > 50%（10次统计） → 熔断30秒 → 半开探测
    熔断期间走降级逻辑

队列削峰（MQ）:
  高峰期订单支付回调 → MQ → 异步消费
  避免瞬间大量 DB 写入
```

### 6.2 安全设计

**参照 RAG 检索到的往年概要设计文档：**
> "数据传输：全站 HTTPS 加密；用户认证：JWT 令牌机制；密码存储：BCrypt 加密算法；SQL 注入防护：使用参数化查询；XSS 防护：前端输入验证与过滤"

本系统在此基础上进一步细化：

| 安全类型 | 实现方式 |
|---------|---------|
| **传输安全** | TLS 1.2+，HSTS 头部，禁止 HTTP 明文 |
| **认证安全** | JWT + Redis 双重校验，Token 登出即失效 |
| **密码安全** | bcrypt（cost=12），禁止明文，禁止 MD5/SHA1 |
| **SQL 注入防护** | MyBatis 参数化查询（#{param}），禁止字符串拼接 SQL |
| **XSS 防护** | 前端对用户输入 HTML 转义；后端对返回内容进行 encode |
| **CSRF 防护** | 移动端天然防 CSRF；Web 端使用 SameSite Cookie 或 CSRF Token |
| **支付安全** | 金额服务端计算，回调验签，幂等处理防重复扣款 |
| **敏感数据脱敏** | 手机号展示 138\*\*\*\*5678；支付信息不落本地库 |
| **接口权限** | RBAC 注解校验，越权访问返回 40102 |
| **日志脱敏** | 日志中手机号/密码/Token 自动脱敏（日志拦截器） |
| **文件上传** | 仅允许 jpg/png/webp，大小限制 5MB，存 OSS 不存本地 |
| **依赖安全** | 定期执行 `mvn dependency:check` 扫描已知 CVE 漏洞 |
