## 2 系统需求概述

### 2.1 需求规定

系统面向四类角色，核心输入输出如下：

| 角色 | 主要输入 | 主要输出 | 关键处理 |
|------|---------|---------|---------|
| **学生用户** | 搜索关键词、下单信息、支付指令、评价内容 | 店铺/商品列表、订单状态、配送位置 | 推荐排序、下单支付、实时追踪 |
| **商家** | 商品信息、订单操作指令、备餐状态更新 | 订单通知、销售数据报表 | 接单流程、数据汇总 |
| **骑手** | 接单/抢单指令、位置上报、状态更新 | 派单信息、导航路线 | 智能派单算法、路径规划 |
| **管理员** | 审核操作、下架指令、评价处理 | 平台数据总览、审核结果 | 内容治理、权限管控 |

**核心性能指标**（来自 SRS NFR-020）：
- API 平均响应时间 ≤ 1秒，P95 ≤ 2秒
- 高峰期支持并发用户数 ≥ 1,000
- 系统年可用性 ≥ 99.5%
- 订单创建成功率 ≥ 99.9%
- 实时消息推送延迟 ≤ 10秒

### 2.2 业务目标

1. **便捷点餐**：为校园学生提供专属外卖下单入口，全流程（浏览→下单→追踪）≤ 5分钟
2. **高效接单**：商家在5秒内收到新订单推送，支持一键接单与状态流转
3. **智能配送**：骑手智能派单响应 ≤ 3秒，全程状态实时同步
4. **平台治理**：管理员可对商家、菜品、评价进行全面审核与管控
5. **增值运营**：优惠券、拼单、美食榜单提升用户活跃度与平台黏性

### 2.4 设计约束

| 约束项 | 说明 |
|--------|------|
| **架构约束** | 前后端严格分离，后端提供 RESTful API，移动端与 Web 端共用同一套接口 |
| **技术栈约束** | 后端以 Java（Spring Boot）为主，数据库为 MySQL + Redis，不引入未经验证的中间件 |
| **安全约束** | 所有网络传输强制 TLS 1.2+；用户密码 bcrypt 加密；支付金额服务端计算不可客户端篡改 |
| **合规约束** | 符合《个人信息保护法》《网络安全法》；食品相关内容符合食品安全法规 |
| **部署约束** | 初期单校区部署，架构预留多校区扩展能力（配置化地理范围） |
| **接口依赖约束** | 地图和支付依赖第三方 API，需做适配层（Adapter Pattern）以便切换供应商 |
| **时间约束** | 分迭代交付：P0 核心功能（点餐+接单+配送）优先，P2 增值功能后续迭代 |

### 2.7 系统总体架构

系统采用**前后端分离 + 分层微服务**的总体架构，分为五个层次：

```
┌────────────────────────────────────────────────────────────────┐
│                        客户端层                                 │
│   [iOS App]  [Android App]  [Web浏览器(PC/平板)]               │
└──────────────────────────┬─────────────────────────────────────┘
                           │ HTTPS / WSS
┌──────────────────────────▼─────────────────────────────────────┐
│                      接入层 (API Gateway)                       │
│   Nginx 反向代理 → 负载均衡 → 路由转发 → 限流 / 鉴权            │
└──────────────────────────┬─────────────────────────────────────┘
                           │ REST / WebSocket
┌──────────────────────────▼─────────────────────────────────────┐
│                      服务层 (Backend Services)                  │
│                                                                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│  │用户服务  │ │商品服务  │ │订单服务  │ │配送服务  │          │
│  │(User)    │ │(Product) │ │(Order)   │ │(Delivery)│          │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
│  ┌──────────┐ ┌──────────┐                                     │
│  │营销服务  │ │管理服务  │                                     │
│  │(Marketing│ │(Admin)   │                                     │
│  └──────────┘ └──────────┘                                     │
└──────────────────────────┬─────────────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────────────┐
│                     中间件层 (Middleware)                        │
│   [Redis 缓存]  [消息队列 MQ]  [消息推送服务]                   │
└──────────────────────────┬─────────────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────────────┐
│                       数据层 (Data Layer)                       │
│   [MySQL 主库] ←──主从复制──→ [MySQL 从库]                     │
│   [Redis 缓存集群]                                              │
└────────────────────────────────────────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────────────┐
│                    外部服务层 (3rd Party)                        │
│   [高德/百度地图 API]  [微信支付/支付宝]  [短信服务]  [CDN]     │
└────────────────────────────────────────────────────────────────┘
```

**架构决策记录（ADR）**：

| 决策项 | 选择方案 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|---------|---------|---------|-----------|
| 后端架构 | 分层微服务（Spring Boot 多模块） | 单体应用 / 全微服务 | 课程周期适中；比单体易于模块化测试；比全微服务运维复杂度低 | 模块间耦合需通过接口层约束 |
| 实时通信 | WebSocket（订单追踪、骑手位置） | 长轮询 / SSE | WebSocket 双向通信延迟最低，适合高频位置更新 | 需处理连接断线重连逻辑 |
| 缓存方案 | Redis | Memcached / 本地缓存 | 支持多种数据结构（List/Hash/ZSet），兼顾会话、限流、排行榜等多场景 | 需做 Redis 持久化配置防数据丢失 |
| 派单消息 | MQ 异步解耦（Redis Pub/Sub 或 RabbitMQ） | 同步 HTTP 调用 | 高峰期订单量大，异步解耦防止服务阻塞 | 需保证消费幂等性 |
| 移动端 | 原生双端（iOS + Android）或 React Native | 纯 Web（H5） | 原生体验更佳（导航、推送）；React Native 可降低双端维护成本 | React Native 部分原生功能需要桥接 |

## 3 系统接口设计

### 3.2 接口设计

#### 3.2.1 用户接口

**移动端（iOS / Android）**：

- 采用 React Native 或原生双端开发，提供与美团外卖风格一致的 Tab 式导航：首页 / 订单 / 我的
- 商家端独立 App 或 Web 管理后台，主界面为订单列表 + 实时通知
- 骑手端独立 App，核心界面为地图 + 订单信息 + 状态操作按钮

**Web 端（管理后台）**：

- 管理员后台采用 PC Web（React + Ant Design），侧边栏导航，包含：商家审核、内容管理、订单管理、数据概览
- 用户 Web 端（H5）支持移动浏览器访问，与 App 功能对齐

**主要页面与跳转关系**：

```
[启动页]
   └→ [登录/注册页]
         └→ [主页] (底部Tab)
               ├→ [搜索页] → [搜索结果]
               ├→ [店铺列表页（筛选）]
               │     └→ [店铺详情页]
               │           └→ [商品详情页] → [购物车]
               │                                └→ [结算页] → [支付页] → [订单追踪页]
               ├→ [订单列表页]
               │     └→ [订单详情/追踪页]
               └→ [个人中心页]
                     ├→ [历史记录页]
                     ├→ [收藏页]
                     └→ [优惠券页]
```

|--------|---------|-----------|---------|------|

| IF-004b | 支付宝 API | HTTPS + SDK | JSON + 签名 | 用户在线支付（备选） |

**内部服务间接口（概要）**：

| 调用方服务 | 被调用服务 | 通信方式 | 说明 |
|-----------|---------|---------|------|
| 订单服务 | 用户服务 | REST HTTP | 验证用户身份与地址 |
| 订单服务 | 商品服务 | REST HTTP | 查询商品价格与库存 |
| 订单服务 | 营销服务 | REST HTTP | 校验并核销优惠券 |
| 订单服务 | 配送服务 | MQ 消息 | 支付完成后异步触发派单 |
| 配送服务 | 用户服务 | REST HTTP | 获取配送地址 |

| 所有服务 | 消息推送服务 | MQ 消息 | 统一推送通知 |

**接口安全规范**：

- 所有外部 API 调用强制 HTTPS（TLS 1.2+）
- 服务间调用通过内网，使用服务内部 Token 鉴权
- 支付接口使用平台提供的签名验证机制，防篡改
- API Gateway 统一处理 JWT 校验、限流（用户维度：100 req/min）

```
User ──[1:1]──> Rider（骑手认证，可选）
User ──[1:N]──> CartItem（购物车条目）
User ──[1:N]──> Order（历史订单）
User ──[1:N]──> Review（评价）
User ──[1:N]──> UserCoupon（优惠券）

Merchant ──[1:N]──> Shop（店铺）
Shop ──[1:N]──> Product（商品）
Shop ──[1:N]──> Order（通过店铺维度统计）

Order ──[1:N]──> OrderItem（订单明细）
Order ──[1:1]──> DeliveryTask（配送单）
Order ──[N:1]──> UserCoupon（使用的优惠券）

DeliveryTask ──[N:1]──> Rider（接单骑手）

GroupOrder ──[1:N]──> Order（拼单内的子订单）

CouponTemplate ──[1:N]──> UserCoupon
```

#### 3.3.2 物理结构设计要点

**关键存储设计**：

| 数据 | 存储位置 | 访问方式 | 说明 |
|------|---------|---------|------|
| 用户基础信息 | MySQL（users 表） | 主键索引 / 手机号唯一索引 | 读多写少，可缓存至 Redis |
| 店铺列表 | MySQL（shops 表） + Redis | 分类/评分复合索引 | 首页推荐结果缓存 TTL=5分钟 |
| 商品信息 | MySQL（products 表） + Redis | shop_id 索引 + 名称全文索引 | 搜索走全文索引，详情走 Redis 缓存 |
| 购物车 | Redis（Hash，key=cart:userId:shopId） | O(1) 读写 | 临时数据，TTL=7天；持久化备份至 MySQL |
| 订单数据 | MySQL（orders + order_items 表） | user_id/shop_id/status 复合索引 | 核心业务数据，事务保证一致性 |
| 骑手实时位置 | Redis（GEO 类型，key=rider:location） | 地理位置查询 | 高频写入，TTL=30分钟 |
| 会话 Token | Redis（String，key=token:userId） | O(1) 读写 | TTL=7天，登出即删除 |
| 限流计数器 | Redis（String，INCR + TTL） | 原子操作 | 接口级别限流 |
| 消息推送队列 | Redis Pub/Sub 或 MQ | 发布订阅 | 订单状态变更事件 |
| 操作日志 | MySQL（admin_logs 表） | created_at 时间索引 | 仅写不改，可后期归档 |

**索引策略**：

| 表名 | 索引字段 | 索引类型 | 用途 |
|------|---------|---------|------|
| users | phone | UNIQUE | 手机号登录/注册去重 |
| shops | (type, rating) | COMPOSITE | 店铺筛选排序 |
| shops | location | 全文/地理 | 按位置筛选（可扩展空间索引） |
| products | shop_id | INDEX | 获取店铺商品列表 |
| products | name | FULLTEXT | 商品名称模糊搜索 |
| orders | (user_id, status) | COMPOSITE | 用户订单列表查询 |
| orders | (shop_id, created_at) | COMPOSITE | 商家订单管理 |
| order_items | order_id | INDEX | 订单明细查询 |
| delivery_tasks | (rider_id, status) | COMPOSITE | 骑手当前任务查询 |
| reviews | (target_type, target_id) | COMPOSITE | 商品/店铺评价列表 |

## 4 系统数据库设计

### 4.1 数据库总体方案

采用 **MySQL 8.0** 作为主关系型数据库，**Redis 6.0+** 作为缓存与实时数据库，**MySQL 主从复制** 实现读写分离。

```
写请求 ──→ [MySQL 主库] ──主从复制──→ [MySQL 从库 1]
                                  └──→ [MySQL 从库 2]
读请求 ──→ [Redis 缓存] ──缓存未命中──→ [MySQL 从库]
```

### 5.2 运行控制

**启动顺序**：
1. MySQL 主从就绪（健康检查通过）
2. Redis 集群就绪
3. 消息队列服务就绪
4. 各后端微服务启动（依赖注册中心检测 DB/Redis 连接）
5. WebSocket 网关启动
6. Nginx 启动，开始接受外部流量

**关键业务流程控制**：

**下单支付时序（含异常路径）**：
```
用户端 → [POST /orders] → order-service
    ├── 查询商品价格（→ product-service）
    ├── 校验优惠券（→ marketing-service）
    ├── 计算金额（服务端）
    ├── 创建订单（status=待支付）写入 MySQL
    ├── 返回支付参数 → 用户端调起支付 SDK
    │
    支付结果异步回调 → order-service
    ├── 验签（防篡改）
    ├── 幂等检查（防重复处理）
    ├── 更新订单 status=已支付（MySQL 事务）
    ├── 核销优惠券（→ marketing-service）
    └── 发布"订单已支付"消息 → MQ
              └── 消费方：配送服务创建 DeliveryTask
                          商家服务推送订单通知
```

**骑手位置上报频率控制**：
- 骑手配送中每10秒上报一次 GPS 坐标
- 服务端写入 Redis GEO，同时通过 WebSocket 推送给订单对应用户
- 骑手离线（超过60秒无上报）标记为"位置不可用"

**优惠券防超发控制**：
```
用户领券 → 检查 Redis coupon:stock:{templateId}
    ├── DECR 原子操作（返回值 ≥ 0）→ 领取成功，写入 user_coupons
    └── DECR 失败（返回值 < 0）    → INCR 回滚，提示"优惠券已领完"
```

### 5.3 运行时间

| 场景 | 时段 | 预期负载 | 处理策略 |
|------|------|---------|---------|
| **就餐高峰** | 11:00-13:00，17:00-19:00 | 并发用户 ≥ 1,000，订单 TPS ≥ 50 | 弹性扩容服务副本；Redis 缓存降低 DB 压力 |
| **平峰时段** | 其余时间 | 并发用户 < 200 | 维持最小副本数 |
| **凌晨维护** | 02:00-05:00 | 极低 | 可执行 DB 备份、榜单更新、数据清理 |
| **美食榜单更新** | 每周一 02:00 | 后台批任务 | 定时任务，不影响在线服务 |
| **订单超时扫描** | 每5分钟一次 | 后台任务 | 扫描 status=待支付且超15分钟的订单，自动取消 |

**容量预估**（单校区初期）：

| 数据类型 | 增长速度 | 1年数据量估算 | 存储方案 |
|---------|---------|------------|---------|

| 骑手位置 | 实时，不持久化 | — | Redis，TTL 30分钟 |
| 日志 | 10MB/天 | 约 3.6GB/年 | MySQL + 定期归档 |

## 6 出错处理设计

### 6.1 错误码规范

系统统一使用以下错误码结构：

```json
{
    "code": 40101,
    "message": "Token 已过期，请重新登录",
    "data": null
}
```

**错误码分段规则**：

| 错误码范围 | 类别 | 示例 |
|----------|------|------|
| 200xx | 成功 | 20000=成功 |
| 400xx | 客户端参数错误 | 40001=参数缺失 40002=格式错误 |
| 401xx | 认证/权限错误 | 40101=Token过期 40102=权限不足 |
| 404xx | 资源不存在 | 40401=店铺不存在 40402=商品不存在 |
| 409xx | 业务冲突 | 40901=商品已下架 40902=优惠券已使用 |
| 429xx | 限流 | 42901=请求过于频繁 |
| 500xx | 服务端内部错误 | 50001=数据库异常 50002=第三方服务异常 |

### 6.2 异常处理策略

| 异常场景 | 处理方式 | 用户提示 | 系统动作 |
|---------|---------|---------|---------|
| 支付网关超时 | 等待回调（最长15分钟），不立即报错 | "支付处理中，请稍候" | 定时任务轮询支付结果 |
| 支付网关不可用 | 降级，保留订单15分钟 | "支付服务暂时不可用，请稍后重试" | 告警运维 |
| 地图 API 超时/不可用 | 降级展示文字地址 | "当前无法显示地图" | 记录日志，不中断订单流程 |
| 数据库连接失败 | 切换从库；仍失败则返回 500 | "系统繁忙，请稍后重试" | 告警运维，触发熔断 |
| 骑手位置丢失（>60秒无上报） | 停止推送位置更新 | "当前无法获取骑手位置" | 记录异常事件 |
| 商品下单时已下架 | 拒绝下单 | "部分商品已下架，请重新确认购物车" | 自动清理购物车中的下架商品 |
| 优惠券并发超发 | Redis 原子 DECR 防超发 | "优惠券已领完" | 无需额外处理 |
| 抢单并发冲突 | 数据库乐观锁（version字段）+ Redis 分布式锁 | "抢单失败，已被他人接走" | 释放锁，返回提示 |
| 第三方短信超时 | 重试1次（间隔3秒） | "验证码发送失败，请重试" | 记录日志 |
| Token 过期 | 返回 401 | "登录已过期，请重新登录" | 客户端清除本地 Token，跳转登录页 |

### 6.3 日志策略

| 日志类型 | 记录内容 | 存储位置 | 保留期 |
|---------|---------|---------|--------|
| **访问日志** | 请求路径、响应码、耗时、用户ID | Nginx 日志 | 30天 |
| **业务日志** | 关键业务操作（下单、支付、接单、派单） | 应用日志文件 | 90天 |
| **错误日志** | 异常堆栈、错误码、上下文参数 | 应用日志文件 | 90天 |
| **操作日志** | 管理员操作记录（审核/下架/删除） | MySQL admin_logs 表 | 永久 |
| **支付日志** | 支付请求/回调详情（脱敏） | 独立安全日志 | 5年（合规要求） |

## 7 注解

### 7.1 架构设计说明补充

**为什么不使用全量微服务（Service Mesh）**：
课程项目周期有限，全量微服务（含 Istio、链路追踪、服务网格等）运维复杂度远超团队能力边界。本方案采用"轻量微服务"——多模块单部署单元（Spring Boot Multi-Module），模块间通过内部 HTTP 调用，保持接口隔离，在课程阶段即可平滑演进为独立部署。

**WebSocket 连接管理说明**：
WebSocket 连接（订单追踪、骑手位置）由独立的 `websocket-gateway` 服务管理，通过 Redis Pub/Sub 实现跨服务实例消息广播。骑手位置上报通过 HTTP API → delivery-service → Redis GEO → 发布至 Pub/Sub → WebSocket 网关推送给用户。

**数据一致性保证**：
- 支付完成与订单状态更新：同一 MySQL 事务，确保原子性
- 优惠券核销：事务内执行，防止重复使用
- 骑手抢单并发：Redis 分布式锁（SETNX）+ MySQL 乐观锁双重保障
